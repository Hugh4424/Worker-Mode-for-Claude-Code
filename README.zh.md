## 它为什么诞生:一次真实排查

起因是一句真实的求助:

> **「最近 10 来天,opus 模型用量是 sonnet 的 20 倍……需要帮我调查为什么。」**

根因:**主会话(orchestrator)总是自己读文件、自己改代码**,而不是把活派给子代理——把所有重活灌进了最贵的模型。

然后是三个真实会话砸出来的硬数据:

| 主会话干了什么 | 实测 |
|---|---|
| 委派率 | **只有 1–2%** |
| 自己亲手干活 | **833–1128 次** |
| 单会话上下文膨胀 | **25–37 万 token** → 频繁触发自动压缩 |
| 并行派发 | **0 次** |

> 「我在系统提示词里明白地说了要多派子代理执行任务,但真实执行时,主会话还是一直读文件、改代码,**总是忘记用子代理**。」

**三套历史方案全失败:**

- 🚧 **硬门拦截**(拦住工具)→ 被「分段读」后门绕过,已 revert。
- 🔕 **软提醒**(提醒插件)→ 提醒 43 次,行为**零改变**。
- 📏 **纯协议 + 固定行数硬规则** → 误伤了主会话本该自己读的那些指令文件。

根因有两条:**软提示在长上下文里被注意力稀释**,**硬拦导致「尝试→被拦→绕路」——时间和 token 双重浪费。**

所以这个插件走了没人走过的那条路:**根本不拦截,只改变默认念头。**

## 它做什么

装上它,主会话就会带上**工头念头**:重活、脏活、读量大的活、能并行的活,都派给一队各有专长的小工——工头只拆任务、收摘要、判结果,自己的上下文保持轻量清醒。

```
        装之前                                 装之后
┌─────────────────────────┐         ┌─────────────────────────┐
│  主会话                  │         │  主会话(工头)          │
│  ├─ 读 3 万行大文件      │         │  ├─ 只读轻量状态文件     │
│  ├─ grep × 127           │         │  └─ 派活 ↓               │
│  ├─ git show × 219       │         └──────────┬──────────────┘
│  ├─ Edit × 13            │            ┌────────┼────────┐
│  └─ 上下文: 37万 💥      │         读文件员  实现员  审查员
│     (反复触发压缩)       │         (重读量都在小工那边发生
└─────────────────────────┘          只有摘要回到主会话
   什么都自己干                        主会话上下文始终轻量)
```

<sub>上图「装之前」的数字来自一个真实会话:127 次源码 grep + 219 次 `git show/diff` + 13 次自己改,自:派比 25.6 : 1。</sub>

### 这支队伍(6 个可移植小工 + 1 个工头)

每个小工都带一句 `description`「什么活该派我」——工头读 description 临场判断派谁,**绝不靠硬编码分工表**。加一个新小工、写清它的 description,路由就自然生长。

| 小工 | 什么时候派它 |
|---|---|
| 🔎 `researcher` | 查外部文档、API 参考、多源事实核查 |
| 📖 `file-reader` | 读大文件 / 长日志,太重不该进主会话 |
| 🛠️ `implementer` | 写新代码(自带 TDD 纪律 + YAGNI 意识) |
| 🔬 `reviewer` | 产物被接受前做独立审查 |
| ✅ `qa` | 跑测试、做验收、收可证伪证据 |
| 🩹 `fixer` | 复现 → 找根因 → 修 finding 或失败 |
| 🦺 `coordinator` | 工头本体——带全工具,负责派活 |

## 60 秒上手

**1. 安装** — 把 `Worker-Mode-for-Claude-Code/` 整个目录放进项目的 plugin 位置。

**2. 配好唯一必填项**(像配 API key 一样):

```bash
export WORKER_LOG_PATH=/abs/path/to/worker-log.jsonl
```

> 这是唯一的外部契约。没配 `WORKER_LOG_PATH` 就不记录——而且会**明确报错**,绝不偷偷写到错地方。(但它**永远不会阻断你的会话**。)

**3. 把 agent 接进项目**(一次性,幂等):

```bash
bash .specify/scripts/bash/setup-delegation-workers.sh
```

**4. 正常用 Claude Code 就行。** 主会话会自己派活,每个小工结束时 `SubagentStop` 自动记一笔,你不用做任何额外动作。

## 看它到底有没有效

两个互补的事后 CLI——**一次性、非常驻、任务执行期零负担。**

```bash
# 派出去的活(读 worker-log)
node tools/check-metrics.mjs --log $WORKER_LOG_PATH
node tools/check-metrics.mjs --log $WORKER_LOG_PATH --json

# 工头自己扛了多少本该派出去的活(读它的转录)
node tools/check-context-health.mjs <主会话转录.jsonl>
```

`check-metrics` 看**派出去的**,`check-context-health` 看**留下来的——工头自己扛了、本可委派的那些读量**。两个合起来才是全貌。

**它报告的指标**(全部**仅供观察、绝不当门禁、绝不当可刷的阈值**):

- **委派率** — 派发次数 ÷ 主会话总动作数。
- **上下文** — 拆成两个诚实的数,不用单一首尾差掩盖语义:
  - **净增长**(末值 − 首值;JSON 字段名 `context_net_growth`)— **可以为负**,而且负值是**好信号**:缓存压缩让上下文净收缩,说明委派有效。
  - **峰值**(最大值 − 首值;JSON 字段名 `context_peak`)— 会话过程中最重那一刻的扩张压力,恒 ≥ 0。
- **主会话 vs 小工 token** — 理想形态是重读量都压在小工身上,工头保持轻量。

> 为什么只供观察?指标当门禁会诱导刷分和坏行为。信任的唯一真答案是**重跑 + 签名**,不是一个能被你拉高的数字。

## 为什么是「纯激励、零拦截」?

这是设计的**灵魂**(决策 D1),而且是刻意的:

> 「我希望让 orchestrator 自己**主动愿意**去委派子代理执行任务,我不想靠任何事后强硬阻拦来执行。」

任何「事后才发现走不通」的拦,都会让模型**尝试→被拦→绕路**——浪费在念头那步已经发生了。所以这里**永远没有 PreToolUse 阻断、没有工具限权、没有运行时门禁。** 是否委派、什么时候委派,100% 是工头自己的判断。

**红线——这几类活工头必须自己干,绝不派出去:**

1. 读执行剧本(stage 定义、contract、硬规则、被审对象原文)——*有损摘要派给小工去指挥流程 = 步骤漂移的总根因。*
2. 读进度 / 状态文件——这是工头做调度判断的依据。
3. 需要当前会话上下文的判断。
4. 从当前会话沉淀经验。

> ⚙️ **两个 hook,都不拦截:** `SessionStart` 在 `WORKER_LOG_PATH` 没配时浮出一条**非阻断**提醒;`SubagentStop` 在每个小工**结束后**记一条日志。两者都绝不改变或阻断行为——记录是纯观察者(FR-LOG-006)。

## 完全可移植

零依赖任何宿主平台。没有 task 目录概念、不读任何平台专属状态、不挂任何外部 gate。纯 Node ESM(无 npm 依赖),只用标准 Claude Code hook + agent + 一个环境变量。把目录丢进任意项目、配好 `WORKER_LOG_PATH`、接好 agent——就能用。

---

## 什么时候别用它

诚实建立信任,所以直说:这个插件在**长的、多步的开发会话**里才值回票价——那种上下文膨胀、模型成本浪费是真问题的场景。要是一个快速的一次性问题、一处小改、一个用完即弃的脚本,工头那套仪式不值得,直接干就行。无论哪种它都不会拦你——这正是它的全部要义。
